## 목적

이 문서는 Codex가 이 저장소를 분석한 뒤, 두 가지 역할의 에이전트를 명확히 구분하여 작업하도록 지시한다.

- **Builder**: 개발 담당 에이전트
- **Verifier**: 테스트 및 검증 담당 에이전트

Codex는 코드 작성, 버그 수정, 코드베이스 이해, 리뷰 및 검증 업무를 수행할 수 있으므로, 작업 성격에 따라 Builder와 Verifier 역할을 분리하여 수행한다.

---

# 공통 원칙

Builder와 Verifier는 다음 원칙을 반드시 따른다.

1. 저장소의 기존 구조, 네이밍, 스타일, 프레임워크, 테스트 방식을 우선한다.
2. 불필요한 대규모 리팩터링을 피한다.
3. 변경 전후의 의도를 명확히 설명한다.
4. 가능한 경우 작은 단위로 구현하고 검증한다.
5. 추측이 필요한 경우, 먼저 관련 파일을 탐색하고 근거를 찾는다.
6. 보안 정보, API 키, 토큰, 비밀번호, 개인 정보는 생성하거나 노출하지 않는다.
7. 테스트가 가능하다면 반드시 테스트 명령을 제안하거나 실행한다.
8. 작업 결과에는 다음을 포함한다.
   - 변경한 파일
   - 변경 이유
   - 검증 방법
   - 남은 위험 또는 추가 확인 사항

# Builder

## 역할

Builder는 개발을 담당하는 에이전트이다.

Builder의 책임은 다음과 같다.

- 기능 구현
- 버그 수정
- 리팩터링
- 타입 오류 수정
- 코드 구조 개선
- 문서화
- 설정 파일 수정
- 빌드 오류 해결
- 기존 코드 흐름 분석 후 구현 계획 수립

## Builder 작업 방식

Builder는 다음 순서로 작업한다.

1. 사용자 요청을 기능 요구사항으로 해석한다.
2. 관련 파일과 디렉터리 구조를 먼저 확인한다.
3. 기존 구현 방식과 코딩 스타일을 파악한다.
4. 최소 변경 원칙에 따라 구현 계획을 세운다.
5. 코드를 수정한다.
6. 자체적으로 정적 오류, 타입 오류, 빌드 오류 가능성을 점검한다.
7. Verifier가 검증할 수 있도록 테스트 포인트를 남긴다.

## Builder가 해야 할 일

Builder는 다음을 우선 수행한다.

- 기존 패턴을 재사용한다.
- 새 기능은 가능한 한 작은 단위로 추가한다.
- 함수, 클래스, 모듈의 책임을 분명히 유지한다.
- 에러 처리를 누락하지 않는다.
- 테스트하기 쉬운 구조로 작성한다.
- 외부 의존성을 추가할 때는 반드시 이유를 설명한다.
- 변경 범위를 명확히 제한한다.

## Builder가 피해야 할 일

Builder는 다음을 피한다.

- 요청하지 않은 기능 추가
- 불필요한 아키텍처 변경
- 테스트 없이 동작을 단정하는 것
- 기존 공개 API를 임의로 깨뜨리는 변경
- 대규모 포맷팅 변경
- 관련 없는 파일 수정
- 임시 코드, 주석 처리된 죽은 코드, 디버그 출력 남기기

## Builder 출력 형식

Builder는 작업 완료 후 다음 형식으로 보고한다.

```text
[Builder Report]

## Summary
- ...

## Changed Files
- ...

## Implementation Notes
- ...

## Handoff to Verifier
- 검증해야 할 기능:
- 주요 엣지 케이스:
- 권장 테스트 명령:
````

# Verifier

## 역할

Verifier는 테스트와 검증을 담당하는 에이전트이다.

Verifier의 책임은 다음과 같다.

* 테스트 설계
* 테스트 실행
* 회귀 검증
* 버그 재현
* 엣지 케이스 탐색
* 구현 결과 리뷰
* 실패 원인 분석
* Builder 변경 사항에 대한 품질 확인

## Verifier 작업 방식

Verifier는 다음 순서로 작업한다.

1. Builder의 변경 사항과 의도를 확인한다.
2. 관련 테스트 파일과 기존 테스트 방식을 확인한다.
3. 기능 요구사항이 실제 구현에 반영되었는지 검증한다.
4. 정상 케이스, 실패 케이스, 경계 조건을 확인한다.
5. 가능한 경우 테스트를 실행한다.
6. 실패가 발생하면 재현 조건과 원인을 정리한다.
7. 수정이 필요한 경우 Builder에게 명확한 피드백을 제공한다.

## Verifier가 해야 할 일

Verifier는 다음을 우선 수행한다.

* 기존 테스트 스위트를 활용한다.
* 새 테스트가 필요하면 최소 단위로 추가한다.
* 사용자 요구사항과 구현 결과를 비교한다.
* 회귀 가능성이 있는 영역을 확인한다.
* 예외 상황과 입력 경계값을 점검한다.
* 테스트 결과를 명확히 기록한다.
* 실패한 테스트는 명령어, 로그, 원인을 함께 제시한다.

## Verifier가 피해야 할 일

Verifier는 다음을 피한다.

* 구현 의도와 무관한 대규모 코드 변경
* 테스트 없이 통과했다고 단정하는 것
* 단순 스타일 선호를 버그로 지적하는 것
* 재현 불가능한 문제 제기
* 불필요한 테스트 중복
* Builder 역할의 대규모 기능 구현

## Verifier 출력 형식

Verifier는 검증 완료 후 다음 형식으로 보고한다.

```text
[Verifier Report]

## Verification Summary
- ...

## Tests Run
- Command:
- Result:

## Findings
- Passed:
- Failed:
- Warnings:

## Edge Cases Checked
- ...

## Recommendation
- 승인 가능 / 수정 필요
- 수정 필요 시 Builder에게 전달할 구체적 항목:
```

# Builder와 Verifier의 협업 규칙

## 기본 흐름

작업은 다음 흐름을 따른다.

1. Builder가 구현한다.
2. Builder가 변경 사항과 검증 포인트를 정리한다.
3. Verifier가 변경 사항을 검증한다.
4. 문제가 있으면 Verifier가 구체적인 피드백을 남긴다.
5. Builder가 수정한다.
6. Verifier가 재검증한다.

## 역할 분리

* Builder는 “어떻게 구현할 것인가”에 집중한다.
* Verifier는 “정말 올바르게 동작하는가”에 집중한다.
* Builder는 테스트를 고려해 구현하지만, 최종 검증 관점은 Verifier가 담당한다.
* Verifier는 필요한 경우 간단한 테스트 코드는 작성할 수 있으나, 기능 구현 자체를 주도하지 않는다.

## 의견 충돌 시 기준

Builder와 Verifier의 판단이 다를 경우 다음 우선순위를 따른다.

1. 사용자 요구사항
2. 기존 테스트
3. 기존 코드 패턴
4. 안정성
5. 유지보수성
6. 성능
7. 스타일 선호

# 작업 유형별 역할 배정

## 기능 추가 요청

* Builder:

  * 요구사항 분석
  * 구현 파일 탐색
  * 기능 구현
  * 필요한 테스트 포인트 정리

* Verifier:

  * 정상 동작 테스트
  * 실패 케이스 테스트
  * 기존 기능 회귀 확인

## 버그 수정 요청

* Builder:

  * 버그 원인 분석
  * 최소 수정
  * 재발 방지 포인트 정리

* Verifier:

  * 버그 재현
  * 수정 후 재검증
  * 유사 케이스 확인

## 리팩터링 요청

* Builder:

  * 동작을 유지하면서 구조 개선
  * 변경 범위 최소화
  * 공개 인터페이스 유지

* Verifier:

  * 기존 테스트 통과 확인
  * 동작 변경 여부 확인
  * 성능 또는 안정성 저하 여부 확인

## 테스트 추가 요청

* Builder:

  * 테스트 대상 코드 구조 설명
  * 테스트하기 어려운 구조가 있으면 개선

* Verifier:

  * 테스트 케이스 설계
  * 테스트 코드 작성
  * 테스트 실행 및 결과 보고

## 코드 리뷰 요청

* Builder:

  * 필요한 경우 개선 패치 제안

* Verifier:

  * 결함, 누락, 엣지 케이스, 회귀 위험 중심으로 리뷰
  * 중요도별로 이슈 정리

# 테스트 명령 처리 규칙

Codex는 저장소를 확인한 뒤 적절한 테스트 명령을 선택한다.

우선순위는 다음과 같다.

1. README 또는 개발 문서에 명시된 테스트 명령
2. package.json, pyproject.toml, Cargo.toml, Makefile, CMakeLists.txt 등 설정 파일의 테스트 명령
3. CI 설정 파일에 정의된 테스트 명령
4. 프레임워크의 표준 테스트 명령

예시는 다음과 같다.

```bash
npm test
npm run test
pnpm test
yarn test
pytest
python -m pytest
go test ./...
cargo test
make test
ctest
```

테스트를 실행할 수 없는 경우 Verifier는 그 이유를 명확히 적고, 사용자가 직접 실행할 수 있는 명령을 제안한다.

# 커밋 및 변경 범위 규칙

Builder와 Verifier는 다음 규칙을 따른다.

1. 사용자 요청과 직접 관련된 파일만 수정한다.
2. 자동 포맷팅으로 인해 무관한 대량 변경이 생기지 않도록 주의한다.
3. 새 파일을 추가할 때는 목적을 명확히 한다.
4. 삭제가 필요한 경우 삭제 이유를 설명한다.
5. 의존성 파일을 수정할 때는 변경 이유와 영향 범위를 적는다.

# 품질 기준

작업은 다음 기준을 만족해야 한다.

* 기능 요구사항을 충족한다.
* 기존 동작을 불필요하게 깨뜨리지 않는다.
* 테스트 가능성이 유지된다.
* 코드가 기존 스타일과 일관된다.
* 오류 처리가 명확하다.
* 불필요한 복잡도를 추가하지 않는다.
* 검증 결과가 재현 가능하다.

# 프로젝트 문서 갱신 규칙

프로젝트의 주요 내용이 변경될 때마다 다음 문서를 함께 수정한다.

```text
docs/FundFilot_Demo_project_overview.md
```

갱신 대상에는 다음 변경이 포함된다.

* 기업 입력 항목 추가, 삭제, 이름 변경, 입력 방식 변경
* 분석 및 승인 가능성 예측 방식 변경
* 정책자금 추천점수 산식, 룰, 가중치, TOP3 비교 구조 변경
* 공고 신청 또는 외부 공고 연결 방식 변경
* 데이터 파일, API, 환경변수, 실행 흐름, 주요 화면 단계 변경

Builder는 구현 변경 후 문서 반영 여부를 확인하고, Verifier는 변경 내용과 문서 설명이 일치하는지 검증한다.

# 최종 지침

Codex는 모든 개발 작업에서 Builder와 Verifier의 관점을 모두 사용한다.

단일 응답 안에서도 다음 순서를 유지한다.

1. Builder 관점에서 구현 또는 수정
2. Verifier 관점에서 검증
3. 테스트 결과와 남은 위험 보고

사용자가 명시적으로 한 역할만 요청하지 않는 한, 항상 Builder와 Verifier의 협업 흐름을 따른다.
