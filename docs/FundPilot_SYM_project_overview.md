# FundPilot_SYM 프로젝트 동작 개요

이 문서는 현재 FundPilot_SYM 데모가 `기업입력 -> 분석 -> TOP3 비교 -> 온라인 신청` 흐름에서 어떤 데이터를 받고, 어떤 방식으로 추천점수와 신청 적합도를 계산하며, 사용자를 중진공 정책자금 온라인 신청으로 연결하는지 정리한 문서입니다.

## 1. 프로젝트 목적

FundPilot_SYM은 FundPilot의 프로토타입입니다. 중소벤처기업진흥공단 정책자금 공공데이터와 정책자금별 주요 신청 조건을 활용해 기업 특성에 맞는 정책자금을 추천하고, 추천점수 및 제한/주의 사유를 보여주는 Streamlit 기반 데모 앱입니다.

주요 소스 파일은 다음과 같습니다.

| 파일 | 역할 |
| --- | --- |
| `app/app.py` | Streamlit UI, 입력폼, 추천 룰, 화면 전환, 중진공 온라인 신청 링크 생성 |
| `app/approval_model.py` | 공공데이터 기반 합성 학습데이터 생성 및 RandomForest 신청 적합도 보조 계산 |
| `app/data_store.py` | CSV 원천자료를 SQLite DB로 적재하고 앱/모델에 데이터셋 제공 |
| `app/api_clients.py` | 공공데이터/중소벤처24 API 인증 환경 로딩, ODcloud 및 일반 공공데이터포털 GET 호출, 수집 로그/원문 캐시 연계 |
| `data/*.csv` | 업종별, 매출액별, 업력별 등 중진공 정책자금 지원 실적 데이터 |
| `data/*.pdf` | 기준금리 등 정책자금 분석에 참고하는 원천자료 |

| `docs/input_usage/step1_input_usage.md` | STEP 1 기업정보 입력값이 추천점수, 신청 적합도, 화면 표시에서 어떻게 쓰이는지 정리 |

| `docs/policy_fund/policy_fund_detail.md` | 직접 조사한 정책자금별 상세/요약 조건. 앱 추천 후보와 세부조건 표시의 기준 문서 |

데이터 파일은 프로젝트 루트의 `data/` 폴더에 모아 관리합니다. `app/data_store.py`는 실행 위치와 무관하게 `Path(__file__)` 기준으로 프로젝트 루트를 찾고, 그 아래 `data/`의 CSV 파일을 `data/fundpilot.db`에 적재합니다. `app/app.py`와 `app/approval_model.py`는 CSV를 직접 읽지 않고 `data_store.read_dataset()`을 통해 SQLite DB에서 데이터를 가져옵니다.

### 1-1. 외부 API 가이드 문서

하드코딩된 데이터와 수동 CSV 의존을 Open API 기반 수집으로 줄이기 위해 `docs/api.md`에 API 이름과 가이드 문서 경로를 인덱싱합니다. `docs/api.md`에 등록된 모든 API는 최종 시스템 적용 대상이며, `docs/public_data_api_candidates.md`에 적힌 FundPilot 활용 목적에 맞게 기업입력, 분석, TOP3 비교, 공고신청, 리스크 중 하나 이상의 실제 기능에 연결합니다. 적용 상태는 `docs/api_utilization_matrix.md`에서 API별로 추적합니다. 현재 `docs/api_guides/`에는 중소벤처24 공고, 금융위원회 기업 기본·재무 정보, 중진공 정책자금 지원 현황 계열 API 가이드가 정리되어 있습니다.

Swagger URL만 제공되던 중진공/ODcloud 데이터는 API별 Markdown 가이드로 변환해 저장했습니다. 이 가이드들은 실제 API 키를 포함하지 않고, 인증키는 `DATA_GO_KR_SERVICE_KEY` 같은 환경변수 이름 또는 `{DATA_GO_KR_SERVICE_KEY}` 자리표시자로만 설명합니다.

현재 가이드화된 중진공/ODcloud 계열 데이터는 종업원 규모별, 자산 규모별, 업종별, 업력별, 자금종류별, 권역/지역별, 제조현장스마트화, 이차보전, 청년전용창업, 창업기반지원, 수출기업화, 재창업, 비대면창업, 융자제외 대상 업종 등입니다. 이 데이터들은 추천 점수 보정, 표준 수혜 패턴 산출, 리스크 비교, 융자제외 업종 판정에 사용됩니다.

`docs/api_integration_summary.md`는 위 API들을 개발 순서, FundPilot 단계별 활용, DB 저장 후보, 공통 구현 체크리스트 기준으로 다시 묶은 통합 참고 문서입니다.

API 연동 공통 기반은 `app/api_clients.py`와 `app/data_store.py`에 분리되어 있습니다. `.env`는 `app/.env`와 프로젝트 루트 `.env`를 모두 탐색하며, 현재 사용하는 환경변수 이름은 `SMES24_OPENAPI_TOKEN`, `DATA_GO_KR_SERVICE_KEY`입니다. `API_ENDPOINT_SPECS`는 `docs/api.md`와 `docs/api_utilization_matrix.md`의 모든 API를 `api_registry`에 등록하기 위한 단일 metadata이며, provider, 인증 위치, 인증 파라미터명, paging 방식, 가이드 문서, 저장 후보, 기준일별 endpoint variant를 포함합니다. `fetch_registered_api()`는 이 registry metadata를 기준으로 ODcloud, 일반 공공데이터포털, KOSMES 포털, 중소벤처24 공고 API를 공통 adapter로 호출하고, 전용 저장 함수가 없는 응답은 `api_normalized_records`에 저장할 수 있습니다. ODcloud 계열 API는 `odcloud_get()`이 `page`, `perPage`, `returnType`, `serviceKey` 기본 파라미터로 호출하고, 일반 공공데이터포털 API는 `public_data_get()`이 `pageNo`, `numOfRows`, `resultType=json`, `serviceKey` 기본 파라미터로 호출합니다. KOSMES 포털 형식의 소부장 API는 `kosmes_portal_get()`이 `Key`, `Type=json`, `pIndex`, `pSize` 형식으로 호출합니다. 중소벤처24 공고정보 연계 API는 `fetch_smes24_notices()`가 `token`, `strDt`, `endDt`, `html=no`로 호출하며, `SMES24_OPENAPI_TOKEN` 값은 원본 또는 URL 인코딩 형태 모두 허용하되 호출 직전에 원본 형태로 정규화해 이중 인코딩을 방지합니다. 토큰은 로그와 원문 캐시 파라미터에서 제외합니다. 수집 결과는 `success`, `missing_key`, `http_error`, `failed`, `empty` 상태로 `api_fetch_logs`에 남깁니다. 공고정보 응답은 `external_notices`에 저장될 수 있지만, 현재 STEP 4 화면은 중소벤처24 공고 매칭/검색 링크를 사용하지 않고 중진공 정책자금 온라인 신청 URL로 직접 안내합니다. 융자제외 업종 API는 성공적으로 수집되면 원문 캐시와 별도로 `kosmes_policy_fund_excluded_industries` 정규화 테이블에 저장되고, 앱의 융자제외 업종 판정은 이 API 테이블을 우선 사용한 뒤 기존 CSV 테이블로 대체합니다.

KOSMES 통계 API는 `KOSMES_STAT_API_SPECS` metadata에 데이터셋 키, API명, 기준일별 endpoint, dimension 유형, 가이드 문서를 등록합니다. `ensure_default_api_registry()`는 이 metadata를 `api_registry`에 등록하고, `fetch_kosmes_support_statistics()` 및 `fetch_all_kosmes_support_statistics()`는 명시적으로 호출될 때만 네트워크를 사용해 원문 캐시와 `kosmes_support_statistics` 정규화 테이블을 갱신합니다. 추천/분석 로직은 `read_kosmes_support_statistics()` 또는 `data_store.read_kosmes_support_statistics()`로 dataset, 기준일, 지역, 업종, 업력, 종업원규모, 자산규모, 자금종류 기준 조회를 사용할 수 있습니다.

분석/추천 로직은 완전 자동 수집 전이라도 `kosmes_support_statistics` 또는 기존 CSV 기반 테이블에 통계 데이터가 존재하면 읽어 사용합니다. 통계 테이블이 비어 있으면 기존 CSV 기반 점수와 룰만으로 동작합니다.

중진공 누리집 URL을 직접 스크래핑해 정책자금 세부조건을 추출하는 방식은 정확도와 유지보수성이 낮아 시스템에서 제거했습니다. TOP3 비교 화면에는 `중진공 세부조건 갱신` 버튼이 없으며, 앱은 URL HTML 원문이나 추출 DB를 읽지 않습니다. 현재 추천 후보와 상세조건 표시는 `docs/policy_fund/policy_fund_detail.md`의 `요약 내용` 표를 기준으로 생성하며, 자금종류별 융자 현황/저장 API 패턴 근거를 함께 보여주는 방식입니다.

P1 기업정보 보강은 STEP 1 입력 화면에도 연결되어 있습니다. 금융위원회 `getCorpOutline_V2`는 사용자가 기업명 또는 법인등록번호로 `기업정보 조회` 버튼을 누를 때만 `get_corp_outline_v2(corp_name 또는 crno)`로 호출되며, 응답 후보를 사용자가 선택한 뒤 `선택 기업정보 자동입력` 버튼을 눌러야 입력폼에 반영됩니다. 기업명, 사업자등록번호, 창업일, 상시근로자 수, 증권시장 상장 여부를 보수적으로 채우고, 업종명/주요사업/등록시장 정보는 기존 업종 드롭다운을 직접 덮어쓰지 않고 API 추천 업종으로 표시한 뒤 사용자가 `추천 업종 적용`을 눌러야 반영합니다. 선택 후보에 법인등록번호가 있으면 최근 3개년 기준으로 `get_summ_fina_stat_v2(crno, biz_year)`를 호출해 매출, 영업이익, 자산총계, 자본총계, 부채비율 자동입력을 시도합니다. API 응답이 없거나 실패하면 안내 메시지만 표시하고 기존 수동 입력값을 유지합니다.

개인사업자는 기업 형태를 `개인사업자`로 선택했을 때 금융위원회 `getFnafInfo`를 지역/업종 조건으로 조회해 유사군 재무 벤치마크를 표시합니다. 이 API는 개별 사업자 식별 조회가 아니라 집계성 유사군 조회이므로 입력값을 직접 덮어쓰지 않고, 평균 매출 대비 낮은 경우 분석 리스크 문구로만 연결합니다.

## 2. 전체 사용자 흐름

앱은 Streamlit `session_state.step` 값으로 4단계 화면을 제어합니다.

| 단계 | 화면 | 핵심 기능 |
| --- | --- | --- |
| STEP 1 | 기업 정보 입력 | 기업 기본정보, 재무, 신용, 수혜이력, 정책우선도, 희망 자금 입력 |

## 4. STEP 2: 분석 및 추천점수 표시

분석 단계는 크게 두 갈래로 동작합니다.

1. 정책자금 추천 룰 엔진
2. RandomForest 기반 신청 적합도 보조 계산

### 4-1. RandomForest 신청 적합도 보조 계산

신청 적합도 보조 계산은 `app/approval_model.py`의 `predict_fit_score()` 함수를 통해 수행됩니다. 이 값은 실제 심사 확률이 아니라 공공 집계 데이터와 synthetic 학습 데이터를 기반으로 한 내부 보조 점수입니다.

모델 학습 방식은 다음과 같습니다.

| 단계 | 설명 |
| --- | --- |
| 공공데이터 로드 | SQLite DB에 적재된 업종별, 업력별, 매출액별, 지역별 지원 실적과 저장된 종업원규모별·자산규모별 지원 현황 테이블을 읽음 |
| 가중치 생성 | 각 업종/업력/매출/지역/종업원규모/자산규모 구간의 지원금액 비율을 계산하고, 저장 데이터가 없으면 fallback 분포 사용 |
| 합성 학습데이터 생성 | 공공데이터 분포와 앱 입력 구간을 반영해 약 10,000건의 synthetic 데이터를 생성 |
| 규칙 기반 라벨 생성 | 공공데이터 가중치, 종업원규모, 자산규모, 지역, 특화 태그, 정책 목적 일치, 기술 보유, 제조업, 청년, 체납·신용 제한을 명시적으로 합산해 적합/부적합 라벨 생성 |
| 모델 학습 | `RandomForestClassifier`로 적합도 분류 모델 학습 |
| 확률 예측 | `predict_proba()`의 적합 클래스 확률을 계산한 뒤 저장된 KOSMES 종업원규모, 자산규모, 지역, 특화자금 API 패턴이 있으면 소폭 보정해 0~100%로 반환 |

모델 입력값은 다음과 같습니다.

| 입력값 | 설명 |
| --- | --- |
| 업종 | 금속, 기계, 전기, 전자 등 내부 인코딩 |
| 매출 규모 | 5억 미만부터 300억 이상까지 구간 인코딩 |
| 업력 | 창업일에서 계산한 업력 구간 |
| 자금 목적 | 운전자금, 시설자금, 창업자금, 기술개발 등 |
| 기업 형태 | 예비창업자, 개인사업자, 법인사업자, 소상공인, 중소기업 |
| 수출 단계 | 내수기업, 수출 준비, 수출 10만 달러 미만/이상 |
| 종업원 규모 | 상시근로자 수를 5인 미만부터 300인 이상까지 구간화 |
| 자산 규모 | 자산총계를 5억 미만부터 300억 이상까지 구간화 |
| 지역 | STEP 1 지역 선택값을 지역 feature로 인코딩 |
| 특화 태그 | 청년, 스마트공장, Net-Zero, 혁신성장, 수출, 소부장 여부를 0/1 feature로 인코딩 |
| 기술 보유 여부 | 특허, R&D, 인증, IP 등으로 판정 |
| 제조업 여부 | KSIC 또는 체크값으로 판정 |
| 대표자 연령 | 청년창업 조건 반영 |
| 체납 여부 | 결격 요인 |
| 신용제한 여부 | 결격 요인 |
| 종업원 수, 자산총계, 지역, 특화 태그 | 모델 feature로 사용하며, 저장된 KOSMES API 패턴이 있을 때 신청 적합도 보조 보정에도 사용 |

### 4-2. 신청 적합도 보정

RandomForest 결과는 `app/app.py`의 `adjust_application_fit_score()`에서 한 번 더 보정됩니다.

| 조건 | 보정 방식 |
| --- | --- |
| 하드 블록이 1개 이상 | 신청 적합도 상한 20% |
| 재무 제한 위험 | 신청 적합도 상한 45% |
| 소상공인 제한 가능성 | 신청 적합도 상한 60% |
| 비수도권 | 가산 |
| 신규 고용 10인 이상 | 가산 |
| 3개년 매출 성장률 20% 이상 | 가산 |
| 중점지원분야 | 가산 |
| 기술 보유 | 가산 |
| 저장된 KOSMES 패턴 API | 종업원규모, 자산규모, 지역 실행률, 특화자금 태그가 있으면 신청 적합도에 최대 ±8%p 보조 보정 |

### 4-3. 승인 가능성 예측 모델 보류

현재 시스템은 `승인가능성` 또는 `승인 가능성`이라는 이름의 실제 심사 확률 예측 모델을 운영하지 않습니다. 정책자금별 실제 승인/탈락 이력과 탈락 사유 데이터가 확보되지 않았기 때문에, RandomForest 결과를 실제 승인 확률로 해석하면 안 됩니다.

승인 가능성 예측 모델을 학습하려면 다음 데이터가 추가로 필요합니다.

| 필요한 데이터 | 필요한 이유 |
| --- | --- |
| 개별 신청기업의 입력값 | 기업 단위 feature와 실제 입력 항목을 연결해야 함 |
| 신청한 정책자금명과 세부사업명 | 자금별 심사 기준과 제한 조건이 다르기 때문 |
| 신청금액과 자금 용도 | 한도, 용도, 상환조건 적합성을 판단해야 함 |
| 심사 결과 | 승인/탈락/보완 같은 supervised label이 필요함 |
| 탈락 또는 보완 사유 | 리스크 요인과 제한 사유를 학습하기 위해 필요함 |
| 대출 실행 여부 | 추천 이후 실제 실행 가능성을 검증하기 위해 필요함 |

따라서 현재 시스템은 실제 승인 가능성 예측 모델을 보류하고, 공공 집계 데이터와 명시적 synthetic label 규칙으로 만든 내부 보조값을 `신청 적합도`로만 유지합니다. STEP 2 분석결과 화면의 대표 수치는 최우선 추천 자금의 `추천점수`이며, 화면의 대형 숫자와 게이지는 TOP3 비교에서 사용하는 동일한 `추천점수`를 0~100점 기준으로 보여줍니다. STEP 2 추천 분석 결과 영역에는 추천점수 산정 근거를 표시하지 않고, 상세 근거는 STEP 3 TOP3 비교에서 별도 확장 영역으로 제공합니다.

## 5. STEP 3: TOP3 비교 및 추천점수 부여

TOP3 추천은 `docs/policy_fund/policy_fund_detail.md`의 `요약 내용` 표에서 읽은 정책자금 후보 목록을 순회하되, 각 후보가 `중소벤처기업진흥공단_정책자금 자금종류별 융자 현황`의 실제 `구분` 값과 매칭되는지 함께 확인합니다. 현재 추천 후보는 해당 문서의 요약 표에 있는 창업기반지원자금(일반), 청년전용창업자금, 개발기술사업화자금, 혁신성장지원, Net-Zero 유망기업 지원, 제조현장스마트화, 신시장진출, 긴급경영안정, 재도약, 매출채권팩토링, 동반성장네트워크론 계열 자금입니다. 지원대상, 융자방식, 한도, 금리, 기간, 주요 조건/비고, 출처는 URL 스크래핑 결과를 사용하지 않고 해당 문서의 `요약 내용` 표를 표시합니다.

### 5-1. 추천점수 구성

각 정책자금의 최종 추천점수는 세 요소를 조합합니다.

| 요소 | 함수 | 설명 |
| --- | --- | --- |
| 과거 지원 패턴 점수 | `get_historical_score()` | SQLite DB에 적재된 공공데이터에서 업종·매출·업력별 지원금액 분포를 읽어 계산 |
| 목적/정책 적합 점수 | `purpose_match_score()` | 희망 자금 종류, 기술, 수출, 스마트공장, 재도약 등 매칭 가산 |
| 자격 조건 판정 | `check_required()` | 필수 조건 충족, 제한 사유, 주의사항을 판정 |
| 저장 API 패턴 보정 | `api_pattern_context()` | 종업원규모, 자산규모, 자금종류 전환율, 지역 실행률, 청년/스마트공장/Net-Zero/혁신성장/수출/소부장 특화자금 테이블이 있으면 추천점수와 패턴 리스크 설명에 반영 |

`recommend_fund()`는 TOP3 정렬에 쓰는 최종 추천점수 산식은 유지하면서, 화면 설명을 위해 `과거지원패턴점수`, `목적정책적합점수`, `자격우대충족건수`, `기본추천점수`, `API보정`, `API근거`, `패턴리스크`, `추천점수근거`, `추천점수상세근거`를 결과 DataFrame에 함께 보존합니다.

### 5-2. 과거 지원 패턴 점수

과거 지원 패턴 점수는 다음 비중으로 계산됩니다.

```text
과거 지원 패턴 점수 = 업종 점수 40% + 매출 규모 점수 30% + 업력 점수 30%
```

각 점수는 정책자금 키워드와 공공데이터의 `구분` 컬럼을 매칭한 뒤, 해당 지원금액이 데이터 내 최소~최대 범위에서 어느 정도 위치하는지 0~100으로 정규화합니다.

저장 API 패턴 보정은 기존 과거 지원 패턴 점수를 대체하지 않고 최종 점수에 최대 -10점에서 +14점 범위로 더합니다. 종업원규모/자산규모 구간 점수, 자금종류별 지원결정·대출 전환율, 지역 신청-추천-대여 실행률이 낮으면 `수혜 패턴 이탈 가능` 주의사항으로 표시하고, 높으면 충족조건/API 근거에 표시합니다.

### 5-3. 목적/정책 적합 점수

다음 조건들이 목적/정책 적합 점수에 가산됩니다.

| 조건 | 효과 |
| --- | --- |
| 희망 자금 종류가 정책자금 선호 목적과 일치 | 가산 |
| R&D 투자자금과 기술개발 목적 일치 | 가산 |
| 기술 보유 + 기술 관련 자금 | 가산 |
| 수출기업 + 신시장진출 자금 | 가산 |
| 제조업 + 청년전용창업자금 | 가산 |
| 중점지원분야 | 가산 |
| 비수도권 | 가산 |
| 신규 고용 10인 이상 | 가산 |
| 매출 성장률 20% 이상 | 가산 |
| 수출실적 50만 달러 이상 + 신시장 자금 | 가산 |
| IP 보유 + 기술/혁신 자금 | 가산 |
| 스마트공장 도입 + 스마트화 자금 | 가산 |
| 사업전환/재창업 + 재도약 자금 | 가산 |
| 희망금액 5억 이하 | 소액 신청 가산 |

### 5-4. 추천상태 결정

각 자금은 제한 사유와 적합 점수에 따라 세 가지 상태 중 하나로 분류됩니다.

| 상태 | 조건 | 점수 산식 |
| --- | --- | --- |
| 우선 추천 | 제한 사유 없음 | `과거점수 * 0.7 + 목적점수 + 충족조건수 * 3`, 최대 100 |
| 조건부 검토 | 제한 사유 1~2개이고 목적점수 있음 | `과거점수 * 0.45 + 목적점수`, 최대 70 |
| 조건 불충족 | 제한 사유가 많거나 적합성이 낮음 | `과거점수 * 0.25`, 최대 40 |

이후 추천상태는 배지와 제한 사유 설명으로 유지하되, TOP3 순위 자체는 전체 정책자금을 최종 `추천점수` 내림차순으로 정렬해 상위 3개만 화면에 표시합니다. 동점이면 기존 정책자금 목록 순서를 유지합니다.

## 6. STEP 4: 공고 신청 연결

공고 신청 단계는 실제 접수 제출을 대행하지는 않지만, 추천된 정책자금별로 중진공 정책자금 온라인 신청 화면으로 이동할 수 있는 버튼을 제공합니다. 기존 중소벤처24 공고정보 연계 API 동기화와 검색 링크 fallback은 STEP 4 화면에서 사용하지 않습니다.

### 6-1. 온라인 신청 이동 방식

STEP 4는 Streamlit의 `st.link_button()`을 사용해 모든 추천 자금 카드에 동일한 중진공 정책자금 온라인 신청 URL을 표시합니다.

연결 URL은 다음과 같습니다.

```text
https://digital.kosmes.or.kr/dh/PLFD/APPLY/PSTEP000M0.do
```

화면에서는 다음 버튼을 제공합니다.

```text
중진공 정책자금 온라인 신청으로 이동
```

사용자가 버튼을 누르면 중진공 정책자금 온라인 신청 화면으로 이동합니다. 신청 가능 기간, 접수 상태, 제출 서류는 외부 신청 화면에서 최종 확인해야 합니다.

### 6-2. 자금종류별 융자 현황 표시

추천 결과는 `중소벤처기업진흥공단_정책자금 자금종류별 융자 현황` 데이터의 실제 자금명을 함께 참고합니다. 앱은 `read_dataset("정책자금 자금종류별 융자 현황")`로 로드한 데이터의 `구분` 컬럼을 기준으로 추천 후보의 자금명·분류·검색키워드·광역 키워드와 매칭되는 실제 자금명을 `자금종류 데이터` 근거로 표시합니다.

이 데이터는 이미 추천점수의 `자금종류 전환율` API 보정에도 사용됩니다. 화면에는 점수 보정과 별도로 실제 데이터에 존재하는 자금명 매칭 근거를 표시해, 문서 기반 추천 후보가 공공데이터의 정책자금 명칭과 어떻게 연결되는지 확인할 수 있게 합니다.

### 6-3. 현재 한계

현재 공고 신청 단계는 중진공 외부 신청 화면으로 이동시키며, 실제 신청서 제출이나 서류 업로드를 앱 안에서 처리하지 않습니다. STEP 4에서 중소벤처24 인증 토큰을 사용하지 않으며, 공고별 접수기간·신청 URL·필요 인증을 자동 표시하지 않습니다.

| 현재 구현 | 향후 확장 가능 방향 |
| --- | --- |
| 중진공 정책자금 온라인 신청 URL 연결 | 신청 준비 체크리스트와 필수 서류 안내 |
| 추천 자금별 자금종류 데이터 매칭 근거 표시 | 공식 공고/신청 화면의 세부사업명과 더 정밀한 매핑 |
| 외부 신청 화면에서 접수상태 최종 확인 | 신청서 작성/제출 시스템과의 별도 연계 |

## 7. 현재 추천 로직의 성격

현재 FundPilot_SYM의 추천은 완전한 실제 심사 모델이 아니라, 데모 목적의 하이브리드 방식입니다.

| 구성 | 성격 |
| --- | --- |
| 공공데이터 기반 점수 | 실제 지원 실적 분포를 활용한 패턴 점수 |
| 정책자금 조건 룰 | 주요 융자 제한/우대 조건을 코드로 반영 |
| RandomForest 모델 | 공공데이터 분포와 정책 심사 규칙을 반영한 합성 데이터 기반 예측 |
| 신청 적합도 보정 | 하드 블록과 우대 조건을 반영한 내부 보조 후처리 |
| 신청 연결 | 중진공 정책자금 온라인 신청 화면으로 이동하는 링크 방식 |

따라서 화면의 분석 결과는 실제 신청 자격 확정이 아니라, 기업이 어떤 정책자금에 가까운지 빠르게 탐색하기 위한 참고용 추천 결과입니다. 실제 신청 전에는 중진공 공식 공고와 세부 제한기업 기준을 반드시 확인해야 합니다.
