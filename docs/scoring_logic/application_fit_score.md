# 신청 적합도 계산 방식

이 문서는 FundFilot_Demo의 내부 `신청 적합도` 계산이 어떤 입력을 받고, 어떤 순서로 동작하는지 정리합니다.

이전에는 이 값을 실제 심사 확률처럼 오해할 수 있게 표현했지만, 현재 프로젝트에서는 해당 명칭을 사용하지 않습니다. 실제 정책자금별 승인/탈락 이력과 탈락 사유 데이터를 확보하기 어려워, 실제 심사 확률을 학습하는 모델은 보류합니다. 현재 `app/approval_model.py`는 공공 집계 데이터와 규칙 기반 synthetic 데이터를 사용하는 내부 `신청 적합도` 보조 모델입니다.

STEP 2 화면의 대표 수치는 이 값이 아니라 TOP3 추천 결과의 1위 자금 `추천점수`입니다. 신청 적합도는 내부 보조 분석값으로 계산되며, 세션의 `application_fit_score`에 저장됩니다.

## 1. 호출 위치

신청 적합도는 `app/app.py`에서 기업 정보 입력 후 분석을 실행할 때 계산합니다.

```text
app/app.py
└─ predict_fit_score(...)
   └─ app/approval_model.py
      └─ predict_fit_score(...)
```

## 2. 입력값

신청 적합도 함수는 총 15개 입력을 받습니다.

| 입력값 | 의미 | 사용 방식 |
| --- | --- | --- |
| `industry_label` | 업종 구간 | RandomForest feature |
| `sales_label` | 매출 규모 구간 | RandomForest feature |
| `experience_label` | 업력 구간 | RandomForest feature |
| `purpose_label` | 희망 자금 목적 | RandomForest feature |
| `company_label` | 기업 형태 | RandomForest feature |
| `export_label` | 수출 단계 | RandomForest feature |
| `has_tech` | 기술·특허·R&D·인증 보유 여부 | RandomForest feature |
| `is_manufacturing` | 제조업 여부 | RandomForest feature |
| `ceo_age` | 대표자 나이 | RandomForest feature |
| `has_tax_arrears` | 국세·지방세 체납 여부 | RandomForest feature 및 강제 상한 |
| `has_credit_issue` | 신용정보 제한 여부 | RandomForest feature 및 강제 상한 |
| `employee_count` | 상시근로자 수 | RandomForest feature 및 저장 API 패턴 보정 |
| `asset_total` | 자산총계 | RandomForest feature 및 저장 API 패턴 보정 |
| `region_label` | 지역 | RandomForest feature 및 저장 API 패턴 보정 |
| `special_tags` | 청년, 스마트공장, Net-Zero, 혁신성장, 수출, 소부장 태그 | RandomForest feature 및 저장 API 패턴 보정 |

## 3. 모델 feature

RandomForest 모델에 직접 들어가는 feature는 20개입니다.

```text
industry
sales
experience
purpose
company
export_stage
employee_size
asset_size
region
has_tech
is_manufacturing
ceo_age
has_tax_arrears
has_credit_issue
tag_youth
tag_smart_factory
tag_net_zero
tag_innovation_growth
tag_export
tag_materials
```

`employee_count`, `asset_total`은 내부 구간으로 변환되고, `special_tags`는 태그별 0/1 feature로 변환됩니다. 같은 입력은 모델 예측 후 저장된 KOSMES API 패턴이 있을 때 한 번 더 소폭 보정에도 사용됩니다.

## 4. 입력 인코딩

문자열 입력은 `app/approval_model.py`의 mapping으로 숫자화됩니다.

| 구분 | 예시 |
| --- | --- |
| 업종 | 금속, 기계, 전기, 전자, 섬유, 화공, 식료, 정보, 유통, 기타 |
| 매출 규모 | 5억 미만, 5억~10억 미만, 10억~50억 미만, 50억~100억 미만, 100억~300억 미만, 300억 이상 |
| 업력 | 1년 미만부터 20년 이상까지 구간 |
| 자금 목적 | 운전자금, 시설자금, 창업자금, 기술개발, 수출/글로벌, 긴급경영, 사업전환, 재창업, 구조개선 |
| 기업 형태 | 예비창업자, 개인사업자, 법인사업자, 소상공인, 중소기업 |
| 수출 단계 | 내수기업, 수출 준비 중, 수출 10만 달러 미만, 수출 10만 달러 이상 |
| 종업원 규모 | 5인 미만, 5~9인, 10~19인, 20~49인, 50~99인, 100~299인, 300인 이상 |
| 자산 규모 | 5억 미만부터 300억 이상까지 구간 |
| 지역 | 서울, 경기, 인천 등 STEP 1 지역 선택값 |
| 특화 태그 | 청년, 스마트공장, Net-Zero, 혁신성장, 수출, 소부장 |

알 수 없는 값은 기본값으로 대체됩니다. 예를 들어 업종은 `기타`, 매출/업력은 중간 구간, 기업 형태는 `법인사업자`, 자금 목적은 `운전자금`에 가까운 기본값으로 처리됩니다.

## 5. 학습 데이터 생성

실제 정책자금별 승인/탈락 원자료가 아니라 공공데이터 분포를 반영한 synthetic 데이터를 만들어 학습합니다.

1. SQLite DB에 적재된 업종별, 업력별, 매출액별, 지역별 지원 실적을 읽습니다.
2. 이미 프로젝트에 적용된 종업원규모별 지원 현황, 자산규모별 지원 현황 테이블이 있으면 함께 읽습니다.
3. 각 구간의 지원금액 비중을 가중치로 계산합니다.
4. 종업원규모별, 자산규모별 데이터가 비어 있으면 중소기업 정책자금의 일반적인 지원 구간을 반영한 fallback 분포를 사용합니다.
5. 약 10,000건의 synthetic 학습 데이터를 생성합니다.
6. 규칙 기반으로 적합 여부 라벨을 만듭니다.
7. `RandomForestClassifier`를 학습합니다.

학습 샘플 생성에 사용하는 공공/저장 데이터는 다음과 같습니다.

| 데이터 | 학습 반영 방식 |
| --- | --- |
| 업종별 지원 실적 | 업종 feature 샘플링 확률과 라벨 bias |
| 업력별 지원 실적 | 업력 feature 샘플링 확률과 라벨 bias |
| 매출액 규모별 지원 실적 | 매출 feature 샘플링 확률과 라벨 bias |
| 지역별 지원 실적 | 지역 feature 샘플링 확률과 라벨 bias |
| 종업원규모별 지원 현황 | 종업원규모 feature 샘플링 확률과 라벨 bias, 저장 데이터가 없으면 fallback 분포 사용 |
| 자산규모별 지원 현황 | 자산규모 feature 샘플링 확률과 라벨 bias, 저장 데이터가 없으면 fallback 분포 사용 |
| 앱 입력 특화 태그 | 청년, 스마트공장, Net-Zero, 혁신성장, 수출, 소부장 0/1 feature 및 라벨 규칙 |

학습 라벨은 다음 요소를 반영합니다.

| 요소 | 효과 |
| --- | --- |
| 업종/매출/업력 공공데이터 가중치 | 지원 실적이 큰 구간일수록 기본 적합도 증가 |
| 종업원 규모 공공/저장 데이터 가중치 | 저장된 지원금액 분포가 높은 구간일수록 적합도 증가 |
| 자산 규모 공공/저장 데이터 가중치 | 저장된 지원금액 분포가 높은 구간일수록 적합도 증가 |
| 지역별 공공데이터 가중치 | 지역별 지원 실적 분포가 높은 구간일수록 적합도 증가 |
| 종업원 규모 명시 룰 | 10~99인 구간 가산, 5인 미만 및 300인 이상 일부 감점 |
| 자산 규모 명시 룰 | 10억~70억 구간 가산, 5억 미만 및 300억 이상 일부 감점 |
| 비수도권 지역 명시 룰 | 정책 우대 가능성을 반영해 가산 |
| 기술 보유 | 가산 |
| 제조업 | 가산 |
| 대표자 만 39세 이하 | 가산, 창업자금 목적이면 추가 가산 |
| 창업자금 또는 기술개발 목적 | 가산 |
| 수출/글로벌 목적과 수출 태그 | 목적-태그가 일치하면 가산 |
| 스마트공장, Net-Zero, 혁신성장, 소부장 태그 | 특화 정책자금 적합 신호로 가산 |
| 수출 단계 | 수출 준비/실적 단계에 따라 가산 |
| 체납 | 큰 폭 감점 |
| 신용 제한 | 큰 폭 감점 |
| 소상공인 | 일부 감점 |

라벨은 위 항목을 합산한 0~100점대 내부 score를 만든 뒤, `score / 100` 확률로 `suitable=1`을 부여합니다. 체납 또는 신용 제한이 있으면 큰 폭 감점 후 최종 score 상한을 35점으로 제한합니다. 즉 synthetic label은 실제 승인 여부가 아니라, 현재 확보 가능한 공공 집계 패턴과 정책자금 기본 제한·우대 규칙을 명시적으로 섞은 보조 학습 라벨입니다.

## 6. 예측 및 1차 보정

`predict_fit_score()`는 학습된 RandomForest의 `predict_proba()`로 적합 클래스 확률을 계산합니다. 이 값은 실제 심사 확률이 아니라 synthetic 학습 데이터 기준의 신청 적합도입니다.

```text
기본 신청 적합도 = RandomForest 적합 클래스 확률 * 100
```

체납 또는 신용 제한이 있으면 25% 상한을 먼저 적용합니다.

```text
체납 또는 신용 제한 있음 -> min(예측값, 25.0)
```

체납·신용 제한이 없으면 저장된 KOSMES API 패턴으로 보정합니다.

| 보정 입력 | 참조 데이터 |
| --- | --- |
| 종업원 수 | 종업원규모별 지원 현황 |
| 자산총계 | 자산규모별 지원 현황 |
| 지역 | 지역/권역별 지원 현황 |
| 특화 태그 | 청년, 스마트공장, Net-Zero, 혁신성장, 수출, 소부장 관련 저장 API 테이블 |

이 보정은 최대 `-8%p ~ +8%p` 범위로 제한됩니다.

## 7. app.py 후처리 보정

`app/app.py`의 `adjust_application_fit_score()`가 한 번 더 보정합니다.

| 조건 | 처리 |
| --- | --- |
| 하드 블록 1개 이상 | 최대 20% |
| 재무 제한 위험 | 최대 45% |
| 소상공인 제한 가능성 | 최대 60% |
| 비수도권 | 제한이 없을 때 +3%p |
| 신규 고용 10인 이상 | 제한이 없을 때 +5%p |
| 3개년 매출 성장률 20% 이상 | 제한이 없을 때 +4%p |
| 중점지원분야 | 제한이 없을 때 +5%p |
| 기술 보유 | 제한이 없을 때 +3%p |

최종값은 `0.0 ~ 97.0` 범위로 제한됩니다.

## 8. 현재 화면 표시와의 관계

현재 STEP 2 분석결과 화면은 이 신청 적합도 값을 대표 지표로 표시하지 않습니다.

화면의 대형 숫자와 게이지는 TOP3 추천 결과의 1위 자금 `추천점수`를 보여줍니다. 신청 적합도는 내부 보조 분석값으로만 계산됩니다.

## 9. 승인 가능성 모델 보류 사유

실제 승인 가능성 예측 모델은 현재 보류합니다.

필요한 학습 데이터는 다음과 같습니다.

| 필요한 데이터 | 이유 |
| --- | --- |
| 개별 신청기업의 입력값 | 기업 단위 feature가 있어야 함 |
| 신청한 정책자금명 | 자금별 심사 기준이 다르기 때문 |
| 신청금액과 자금 용도 | 한도와 용도 적합성 판단에 필요 |
| 심사 결과 | 승인/탈락/보류 같은 supervised label 필요 |
| 탈락 또는 보완 사유 | 모델이 리스크와 제한 사유를 학습하는 데 필요 |
| 대출 실행 여부 | 추천 이후 실제 실행 가능성 검증에 필요 |

현재 확보 가능한 중진공 공공데이터는 주로 업종, 업력, 매출, 지역, 자금종류별 집계 통계입니다. 이는 수혜 패턴과 추천점수 보정에는 사용할 수 있지만, 개별 기업의 실제 심사 확률을 학습하기에는 부족합니다.

따라서 현재 시스템은 다음 원칙을 따릅니다.

```text
실제 승인 가능성 예측 모델: 보류
내부 보조 모델 명칭: 신청 적합도
사용자 대표 지표: 추천점수
```
